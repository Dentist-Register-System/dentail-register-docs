Register — System Decision Trees

Where does each functionality go to answer a question — one engine, or a scavenger hunt across the codebase? This is the picture of the temple: the one-stop, honest map of every core system and where its decisions actually live.

Source of truth: the code itself — dentist-registry-backend (engines) + dentist-registry-frontend (the client that must only ask). Permission matrices below are computed by executing the actual code. Roaming maps cited file:line. Snapshot 2026-06-30.

The bar:   User → "Can I X?"→ [ engine ]→ Answer   One question, one place, one answer.
The Sentinel rule. Every core system gets exactly one engine, and that engine is the only place its decision is made — not two places, not "mostly one." Callers (UI, routes, hooks, components, other engines) only ask; nobody else knows. Code like user.role === DOCTOR outside the engine means the wall is already broken. A CI guard in each repo keeps it that way. This page stays 100% honest so a glance tells us the real layout — and the gaps.

1b · Permissions — the FE wing client still decides · 17 sites

The backend is single-source, but the frontend re-derives permissions client-side instead of asking. GET /api/v1/me returns no capabilities, so 17 origin sites branch on memberships[0].role (6 of them also interpreting clinic settings) — a second, drift-prone copy of the rules living in the client. The messenger pattern already exists in the repo (useMemberCapabilities() + permissions-tab.tsx render purely from allowed/reason_code) but is pointed only at "view another member's permissions," never "what may I do."

→ Target: one useMyCapabilities() reading a new clinic-scoped /me/capabilities; every role=== deleted; a FE sentinel guard fails CI on any client-side permission decision. (kind === "doctor" entity-type checks are not authz — they stay.)

There is no appointment engine and no state machine. The decision tree is implied by inline checks across four files. Each question, and the number of places it visits to get an answer:
