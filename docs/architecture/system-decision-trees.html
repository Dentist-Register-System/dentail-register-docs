1 · Permissions Engine built · not wired yet

We built the engine — permissions/policy.py::can(role, settings, action, is_self, target_role), one pure decider — and proved it reproduces today's rules exactly (282 tests). The matrix below is generated by calling that engine.

1000% honest: the engine is not load-bearing yet. It lives on an open PR (#53), not on main, and no call site calls it. So today a permission question still roams — just less far than appointments — to these five places that each decide independently:
  • members/deps.py::require_role — coarse route gates
  • scheduling/booking.py::can_decide / can_coordinate — approve/reject/cancel
  • scheduling/service.py::can_manage_availability
  • invites/service.py::authorize_invite_mgmt
  • members/capabilities.py::resolve_capabilities — the FE matrix (a second encoding of the same rules)
The migration (next E steps) rewires all five to call can(), then a CI guard forbids deciding permissions anywhere else. After that the badge turns green.

The decision tree (the target)

"Can this user approve this booking?" → can(role, settings, DECIDE_BOOKING, is_self) → ✅ / ❌

Every protected action funnels through the one function. The caller supplies facts (role, settings, is_self, target); the engine makes the decision.

2 · Appointments No engine · splattered

There is no appointment engine and no state machine. The "decision tree" is implied by inline checks scattered across four files. Here is each question, and the number of places it has to visit to get an answer — the rabid-dog map.

🐕 "Can this request be approved / rejected?" → 4 places, 2 of them encoding the same rule twice

scheduling/router.py:227,245calls authorize_decide(...)
booking.py:263 authorize_decidefetches the doctor (doctors.get_doctor) to compute "is this their assigned request?"
booking.py:34 can_decidereads role + settings.allow_staff_approval + is_assigned → the actual rule
members/capabilities.pysecond copy of the same rule, for the FE matrix — can drift from the one above

🐕 "Is this request still actionable (can it transition)?" → 5 places, and they disagree

booking.py:291approve: if status != "pending"
booking.py:324reject: if status != "pending" (re-typed)
booking.py:351cancel: if status != "pending" (re-typed)
booking.py:373resend: if status != "pending" (re-typed)
rules.py:17 is_expiredexpiry rule — but it's checked in only approve (booking.py:293), not the other three

🐕 "Is this slot full?" → 2 different computations that can give different answers

booking.py:96 count_consumerscounts by slot_id — the authoritative write-path answer (drives SlotFullError)
service.py:375-401 list_slotsre-counts by start_datetime for the display "full/available" badge — a separate implementation
rules.py:11 capacitythe capacity number (this part is single-source ✓)
booking.py:130 / :299 / service.py:401the >= capacity comparison, written 3× (one with a -1 self-fudge)

→ The diary can say "available" while booking rejects the same slot as full, the moment start_datetime and slot_id drift.

🐕 "What's available on day X?" → 2 resolvers that already disagree

service.py:284 compute_slotsresolves windows for the day, subtracts blocks, dedups overlaps → what's bookable
reads.py:149 get_day_scheduleits own copy of the window-matching rule that ignores blocks entirely

→ A doctor with a full-day block still shows as available in the day diary.

🐕 "What status is this / what are the valid statuses?" → ~20 magic strings, no definition

models.py:72,98status is a bare String(20) — no enum (scheduling is the only domain without one)
booking.py:467 / reads.py:67the "set of valid statuses" hardcoded as a literal list, independently, twice
The one bright spot: appointment status is written only from booking.py — single-writer is intact. The data discipline is fine; it's the logic that has no home.

What it should be

Today

"Can this be approved?", "is it actionable?", "is the slot full?", "what's available?" each sprint across booking.py + rules.py + service.py + reads.py, re-deriving the answer, sometimes disagreeing.

Target — one appointment engine

"Can this transition?" → appointment state machine → ✅ / ❌

One enum of states, one transition table, one assert_actionable, one occupancy function, one availability resolver. The day-of-clinic states (arrived / completed / no-show — which don't exist yet) extend it instead of adding more roadkill.

3 · Availability No engine · splattered

Same shape as appointments. The slot geometry and capacity number are single-source ✓, but "the doctor's availability on day X" is resolved two different ways (see the appointments "what's available" map above), and the recurring/one-off validity rule is checked in three places (schemas.py:18, service.py:136, service.py:448).

Queued after appointments — same treatment: one resolver, one validity rule, one place.
