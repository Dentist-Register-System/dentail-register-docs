How OLD gets each answer

The "old" column above had to be assembled from six different places:

  • scheduling/booking.py — can_decide, can_coordinate
  • scheduling/service.py — can_manage_availability
  • invites/service.py — authorize_invite_mgmt
  • members/capabilities.py — resolve_capabilities
  • 5× require_role(...) gate constants in routers

How NEW gets each answer

"Can I X?"→ can(role, settings, action, is_self, target)→ ✓ / ✕

One pure function. The caller supplies facts; the engine makes the decision. After wiring, a CI guard forbids deciding permissions anywhere else.

Same 45 answers. Six places vs one. That difference — not the answers — is the whole point.

2 · Appointments No engine · splattered

There is no appointment engine and no state machine. The decision tree is implied by inline checks across four files. Each question, and the number of places it visits to get an answer:

🐕 "Can this request be approved / rejected?" → 4 places, 2 of them encoding the same rule twice

scheduling/router.py:227,245calls authorize_decide(...)
booking.py:263 authorize_decidefetches the doctor to compute "is this their assigned request?"
booking.py:34 can_deciderole + settings.allow_staff_approval + is_assigned → the rule
members/capabilities.pysecond copy of the same rule, for the FE matrix — can drift

🐕 "Is this request still actionable?" → 5 places, and they disagree

booking.py:291 / :324 / :351 / :373approve / reject / cancel / resend each re-type if status != "pending"
rules.py:17 is_expiredexpiry rule — checked in only approve (booking.py:293), not the other three

🐕 "Is this slot full?" → 2 different computations that can disagree

booking.py:96 count_consumerscounts by slot_id — the authoritative write-path answer
service.py:375-401 list_slotsre-counts by start_datetime for the display badge — a separate impl
booking.py:130 / :299 / service.py:401the >= capacity comparison, written 3× (one with a -1 fudge)

→ The diary can say "available" while booking rejects the same slot as full.

🐕 "What's available on day X?" → 2 resolvers that already disagree

service.py:284 compute_slotsresolves windows, subtracts blocks, dedups → what's bookable
reads.py:149 get_day_scheduleits own copy that ignores blocks entirely

→ A doctor with a full-day block still shows as available in the day diary.

One bright spot: appointment status is written only from booking.py — single-writer is intact. The data is disciplined; the logic has no home. (And arrived/completed/no-show don't exist yet — net-new.)
